# 飞书陪伴机器人重写计划书

这份计划书用于指导新的 agent 重写当前项目。目标不是继续堆私有脚本，而是把项目整理成一个可公开、可复用、隐私优先的飞书陪伴机器人框架。

## 当前仓库

- 本地目录：`/Users/xujunshan/Code/github_activity_generator`
- 当前目录名：`github_activity_generator`
- GitHub 远端：`git@github.com:yuaiccc/feishu-companion-bot.git`
- 公开项目名建议：`feishu-companion-bot`

## 重写目标

把当前项目重写成一个通用飞书/Lark 陪伴机器人框架，而不是只服务某一对情侣的私有脚本。

默认仓库必须公开安全：

- 不提交 `.env`
- 不提交真实 profile
- 不提交 `state.json`
- 不提交日志
- 不提交 `memory_data/`
- 不提交恋爱笔记私有逻辑
- 不提交真实地址、token、二维码、通话纪要缓存

机器人定位：它是 owner 的小弟/助手，不是 owner 本人的分身。它可以在 owner 不在时帮忙解释、提醒、整理、搜索、维护记忆，但不能伪装成真人。

## 必须保留的能力

### 1. 飞书长连接本地服务

- 本机运行时走飞书事件长连接，即时响应。
- 群聊只有 @ 机器人时回复。
- 私聊可以直接回复。
- GitHub Actions 只作为本机离线兜底，5 分钟轮询，不应替代本地即时服务。

### 2. GitHub 动态推送

- 支持 public GitHub Events 和 private repo commit polling。
- 必须做幂等：
  - event id 去重。
  - PushEvent 再按 head commit sha 去重，避免同一 commit 被两个来源重复推送。
- 动态文案要给非开发者看得懂：
  - commit：一句中文轻量说明“给陪伴机器人新增了 xx / 修复了 xx”。
  - star：一句中文说明“收藏了一个大概和 xx 有关的项目”。
- 不要给 GitHub 动态套很腻的恋爱总结。

### 3. DeepSeek 总结

- “近期活动”无论是用户主动问，还是 action 自动触发，都要走一次 DeepSeek 轻量总结。
- 总结要克制，一句话说明重点，不要堆情绪。

### 4. 记忆系统

- 记忆必须 profile 隔离：`memory_data/<profile_id>/`。
- 记忆要有可见性字段：
  - `private`：绝不进 prompt。
  - `owner_only`：只给 owner 相关回复使用。
  - `public_to_target`：可对目标用户使用。
- 群聊消息不要靠关键词直接写入记忆。
- 是否值得记忆由 DeepSeek 判断，然后私聊 owner 发确认卡。
- owner 点“记住”才落库；点“不要记”则忽略。
- 做 agentic RAG：先按 embedding 检索候选，再让模型按当前问题重排、选择、压缩，最后按预算拼上下文。
- 本地优先：Ollama embedding 可选，默认 fallback 本地 hash embedding，避免开箱即泄露隐私。

### 5. 上下文管理

- 所有 LLM 调用必须经过统一 `context_manager`。
- 明确预算：
  - 最近聊天
  - 相关记忆
  - 通话或妙记摘要
  - 当前消息
  - 外部搜索结果
- 每次调用要记录实际注入了哪些 section 和字符数，避免 prompt 越拼越大。

### 6. 外部搜索

- 优先本地 DeerFlow。
- 可回退 OpenClaw。
- 搜索结果不要直接丢一堆链接。
- 应先给一段自然中文总结，再补“和当前关系或对话有关的点”。
- 来源只轻量提到，最多必要时给 1 个链接。
- GitHub Actions 环境无法访问本机 DeerFlow/OpenClaw，所以云端兜底不要依赖本地搜索。

### 7. 飞书消息体验

- 长回复先快速反馈“正在输入”。
- 群聊不要发记忆按钮卡片，正常聊天即可。
- 群聊流式可以用普通消息气泡编辑来近似流式。
- 如果要使用飞书原生 CardKit streaming，必须承认它是卡片，不是普通气泡。
- 私聊可以用卡片承载按钮，例如“换个说法 / 继续展开 / 记住 / 不要记”。
- 群聊不应每条消息弹“加入记忆”。

### 8. 飞书文档/恋爱笔记能力

- 私有实现不要进入开源仓库。
- 通用框架只保留扩展接口。
- 如果做文档评论：
  - 只评论当天新增内容。
  - 每天最多 2 条。
  - 不要往正文追加总结，优先做划线评论或段落评论。
  - 要避免重复评论。
  - 机器人可以删除自己不合适的评论。
- 飞书文档接口字段必须查官方文档或 `lark-cli schema`，不要猜。

### 9. 服务健康与延迟观测

- 增加健康自检：
  - 飞书 token
  - 长连接状态
  - DeepSeek
  - GitHub
  - 记忆库
  - Ollama embedding
  - DeerFlow/OpenClaw
  - 本机前台窗口读取
- 增加延迟 span：
  - 读消息
  - 检索记忆
  - 通话纪要
  - 外部搜索
  - DeepSeek 首 token
  - 最终回复发送
- 日志本地打印即可，不需要强行接 LangSmith。

## 如果用 Go 重写

飞书不是没有 Go 支持。可以用官方 Go SDK：`github.com/larksuite/oapi-sdk-go`。

但重写意味着这些都要重新接：

- 长连接事件监听
- `im.message.receive_v1`
- `card.action.trigger`
- 发消息
- 回复消息
- 编辑消息
- 卡片/流式卡片
- 文档评论
- 妙记/会议纪要
- tenant access token 管理
- 错误重试和 rate limit

Go 适合把服务做稳，尤其适合长期运行、清晰模块化、低资源占用。但 Python 生态接 DeerFlow、OpenClaw、本地脚本更方便。

建议新 agent 先选一种路线：

- 稳定服务优先：Go 主服务 + Python 子进程 adapter。
- 快速迭代优先：继续 Python，但重构目录和接口。
- 最折中：Go 负责飞书事件、状态机、幂等、健康；Python 负责搜索、文档私有扩展、实验性能力。

## 实施顺序

### 阶段 1：冻结公开边界

- 清理仓库：README 全中文，移除私有脚本和真实数据。
- `.gitignore` 覆盖 `.env`、日志、`state.json`、`memory_data/`、local extensions。
- profile 模板化，真实人设只放本地未跟踪 profile。

### 阶段 2：重写飞书基础层

- 实现 token、发消息、回复消息、编辑消息、长连接收消息。
- 用官方文档或 SDK schema 校验每个 payload。
- 写 dry-run 模式，避免测试时乱发群消息。

### 阶段 3：实现消息路由

- 群聊：仅 @ 回复。
- 私聊：直接回复。
- action：只兜底。
- passive assist：冷场、时间窗口、同话题冷却、幂等。
- proactive topic：一天最多一次，冷场时 @ 双方。

### 阶段 4：实现 LLM 管线

- prompt profile 化。
- 支持流式首 token。
- 支持 JSON 决策调用：记忆判断、搜索意图、回复策略。
- 长回复先快速反馈，再更新最终内容。

### 阶段 5：实现记忆

- 结构化 schema。
- embedding。
- agentic rerank。
- owner 私聊确认写入。
- 审计面板。
- 清洗、合并、删除。
- 严格可见性过滤。

### 阶段 6：实现 GitHub 动态

- public/private 两个来源。
- event id + head sha 幂等。
- DeepSeek 轻量中文总结。
- 表格展示，时间字段完整显示，手机端不要省略严重。

### 阶段 7：实现外部搜索

- DeerFlow adapter。
- OpenClaw fallback。
- 搜索结果二次总结。
- 不给用户直接扔链接表。

### 阶段 8：实现飞书文档扩展接口

- 通用仓库只留接口。
- 私有恋爱笔记逻辑放 local extension。
- 每日新增内容检测、评论上限、重复检测、评论删除都要有状态记录。

### 阶段 9：测试与验收

单元测试至少覆盖：

- 群聊 @ 判断
- 私聊回复
- GitHub 幂等
- 记忆隐私过滤
- DeepSeek JSON 解析失败 fallback
- 搜索结果格式
- 飞书卡片 action 解析

集成测试至少覆盖：

- dry-run 本地收事件
- 真实飞书发一条私聊
- 真实群聊 @ 一次
- GitHub 动态模拟两来源重复 commit
- 记忆候选私聊确认

提交前固定跑：

```bash
lint
unit tests
dry-run smoke
git diff --check
```

## 最容易踩坑

- 不要把 bot 设计成 owner 分身。
- 不要把真实情侣信息写进开源 profile。
- 不要让群聊每条消息都弹卡片。
- 不要靠关键词写记忆。
- 不要把 GitHub Actions 当主服务。
- 不要忽略 GitHub public/private 双来源重复。
- 不要猜飞书字段，尤其是 CardKit、文档评论、事件订阅。
- 不要把外部搜索结果原样贴链接列表。
- 不要让 prompt 无限膨胀。
- 不要让“正在输入/整理中”留在最终回复里。

## 给新 agent 的一句话任务

请把这个项目重写成一个公开安全、profile 可配置、隐私优先、支持飞书长连接即时回复 + GitHub Actions 兜底 + DeepSeek + 本地记忆 + 本地搜索的通用陪伴机器人框架；所有飞书接口以官方文档/SDK schema 为准，私有关系内容只能作为本地 profile/extension，不进入公开仓库。
